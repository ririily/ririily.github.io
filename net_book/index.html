2 用电信号传输TCP/IP数据#

协议栈内部结构#

应用程序会将收发数据等工作委派给下层的部分来完成，下面是 Socket 库，其中包括解析器，解析器用来向 DNS 服务器发出查询

浏览器、邮件等一般应用程序收发数据时用 TCP； DNS 查询等收发较短的控制数据时用 UDP。

然后是用 IP 协议控制网络包收发操作的部分。在互联网上传送数 据时，数据会被切分成一个一个的网络包 A，而将网络包发送给通信对象的操作就是由 IP 来负责的。此外，IP 中还包括 ICMP 协议和 ARPB协议。 ICMP 用于告知网络包传送过程中产生的错误以及各种控制消息，ARP 用 于根据 IP 地址查询相应的以太网 MAC 地址 。

IP 下面的网卡驱动程序负责控制网卡硬件，而最下面的网卡则负责完 成实际的收发操作，也就是对网线中的信号执行发送和接收的操作。

协议栈是根据套接字中记录的控制信息来工作的。

创建套接字时，首先分配一个套接字所需的内存空间，然后向其 中写入初始状态。

控制信息分为两类

  • 数据包头部信息
  • 套接字中记录的用来控制协议操作的信息

套接字和描述符 套接字接口为网络通信增加了一个新的抽象，即套接字。和文件一样，每个活动的套接字都有一个整数来标识，我们将其称为套接字描述符。

connect（< 描述符 >, < 服务器 IP 地址和端口号 >, …）

上面的调用提供了服务器的 IP 地址和端口号，这些信息会传递给协议 栈中的 TCP 模块。然后，TCP 模块会与该 IP 地址对应的对象，也就是与 服务器的 TCP 模块交换控制信息

连接操作的第一步是在 TCP 模块处创建表示连接控制信息的头部。 通过 TCP 头部中的发送方和接收方端口号可以找到要连接的套 接字。

协议栈并不是一收到数据就发送出去，发送数据有两个判断要素：

  1. 大小。数据大小达到MSS(最大分段大小)时发送以避免大量小数据包
  2. 时间。数据发送频率不高时凑MSS会大大提高网络延迟，因此协议栈中设置了一个定时器，达到一定时间也会把网络包发送出去

这两个要素是互相矛盾的，大小对应网络效率，时间对应网络延迟。

如果仅靠协议栈来判断发送的时机可能会带来一些问题，因此协议栈也给应用程序保留了控制发送时机的余地。应用程序在发送数据时可以指定一些选项，比如如果指定“不等待填满缓冲区直接发送”，则协议栈就会按照要求直接发送数据。像浏览器这种会话型的应用程序在向服务器发送数据时，等待填满缓冲区导致延迟会产生很大影响，因此一般会使用直接发送的选项。

TCP发送的包会进行缓存，如果没有收到某个包的ACK就重发。由于TCP强大的错误补偿机制，网卡、集线器、路由器都没有错误补偿机制，一旦检测到错误就直接丢弃相应的包。

网络的错误检测和补偿机制关键点#
  • ACK等待时间（多久没受到ACK进行重传）。太短在网络阻塞时会进行多余的重传，太长会使包的重传大大延迟。TCP采用动态调整的方法。
  • 使用窗口有效管理ACK包。窗口大小一般由接收方的接收能力决定
三次握手四次挥手（速记版）#

三次握手

  1. 客户端发送SYN
  2. 服务器返回SYN
  3. 客户端发送ACK

四次挥手

  1. 客户端发送FIN
  2. 服务器返回ACK
  3. 服务器发送FIN
  4. 客户端返回ACK
数据收发操作小结#
  • 第一步：创建套接字
  • 客户端向服务器端发起连接
  • 数据收发
  • 断开（一般是由服务器发起，在HTTP1.1中可能是客户端先发起）
包#

TCP/IP数据包包含MAC头部和IP头部

IP 和以太网分工协作，其中以太网的部分也可以替换成其他的东西，例如无线局域网、ADSL、FTTH 等

IP 模块根据路由表 Gateway 栏的内容判断应该把包发送给谁。
